# Researcher Evidence Assembly Design

**Context**

`ResearcherAgent` currently asks the model to output an `EvidencePackage` that includes both open-ended reasoning fields and deterministic retrieval fields. This mixes two different sources of truth:

- model-generated summary fields
- retrieval-system-generated query/chunk fields

That boundary is weak. `RetrievedKnowledgeChunk` is produced by the retrieval tool and should not be regenerated by the model. The same applies to the final `queries` value once the runtime already knows which retrieval call produced the final evidence.

**Goals**

- Keep `EvidencePackage` as the final runtime object consumed downstream.
- Stop asking the model to generate deterministic retrieval fields.
- Make the model output only open-ended researcher summary fields.
- Keep multi-query retrieval represented as multiple `retrieveKnowledge` tool calls, not as a list argument on one tool call.

**Design**

Introduce a model-facing summary class, tentatively `ResearcherSummary`, containing only:

- `evidenceSummary`
- `limitations`
- `uncoveredPoints`

`EvidencePackage` remains the final researcher result returned by `ResearcherAgent`, but its data will be assembled from two sources:

1. **Model output**
   - `evidenceSummary`
   - `limitations`
   - `uncoveredPoints`
2. **Runtime memory**
   - `queries`
   - `chunks`

`queries` and `chunks` will both be derived from the **last executed `retrieveKnowledge` tool interaction**:

- `queries` becomes a single-element list containing the query argument from the last `retrieveKnowledge` call
- `chunks` becomes the parsed `List<RetrievedKnowledgeChunk>` from that same tool result payload

This keeps the two fields semantically aligned: the query shown in the evidence package is exactly the query that produced the attached chunks.

**Why Last Retrieval Instead of Aggregation**

The researcher may probe with intermediate rewrites or branch queries that are low quality, empty, or superseded. Using the last retrieval as the final evidence source matches the current product intent: the final answer should be grounded in the evidence set the agent most recently chose to rely on.

**Tool Shape**

`RetrieveKnowledgeTool` should keep a single `query` argument.

Multi-query retrieval is already naturally expressed by returning multiple `retrieveKnowledge` tool calls in one decision. Changing the tool to accept `queries: []` would complicate pairing tool inputs and outputs, and would make the final “last retrieval query + last retrieval chunks” assembly less clear.

**Prompt Changes**

`ResearcherAgentContextFactory` should explicitly tell the model:

- after the first fixed retrieval, it may decide to:
  - stop
  - rewrite the query and retrieve again
  - emit multiple `retrieveKnowledge` calls for multi-branch retrieval
- final completion must output strict JSON matching `ResearcherSummary`, not `EvidencePackage`

The prompt should no longer imply that the model is responsible for `queries` or `chunks`.

**Runtime Changes**

`ResearcherAgent` should:

1. keep the existing deterministic first retrieval behavior
2. keep the existing model-driven later retrieval behavior
3. on completion:
   - parse `ResearcherSummary`
   - extract the last executed `retrieveKnowledge` argument/result from transcript memory
   - deserialize the result JSON into `List<RetrievedKnowledgeChunk>`
   - assemble a final `EvidencePackage`

If no retrieval result exists, `queries` and `chunks` should fall back to empty lists.

**Testing Strategy**

Update tests to cover:

1. prompt now mentions rewrite/split behavior and `ResearcherSummary`
2. model completion can omit `queries/chunks`
3. final `EvidencePackage` still contains `queries/chunks`, but they come from the last tool interaction in memory
4. evidence contract tests deserialize `ResearcherSummary` separately from `EvidencePackage`

**Review Notes**

- 中文注释应放在 runtime 组装 `EvidencePackage` 的逻辑附近，说明 `queries/chunks` 以最后一次真实检索结果为准，避免模型伪造证据明细。
- 不修改 `RetrieveKnowledgeTool` 为 `queries: []`，因为当前 runtime 已经能通过多个 tool calls 表达多路查询。
