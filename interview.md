# 面试问题整理

## 1. 项目开发时遇到过什么困难？怎么解决的？

这个项目里我遇到的一个核心困难，是怎么让大模型稳定地按照 ReAct 模式工作。

一开始我的实现思路比较直接：我在 Prompt 里要求模型一次回复同时输出 `THOUGHT`、`ACTION`、`OBSERVATION`，希望它自己把推理、动作和观察结果都组织好。这个方案看起来简单，但实际调试时很不稳定。模型经常会出现几个问题：

- 一次回复里输出多个 `ACTION`，导致执行边界不清晰。
- `ACTION` 里的工具调用格式不规范，参数也容易漂移，不方便程序可靠解析。
- 有时候它会把本该由系统执行后再反馈的 `OBSERVATION` 也直接“脑补”出来，整个循环就不够严格。

本质上，这个问题不是业务逻辑难，而是“让模型遵守固定文本协议”这件事很脆弱。只要 Prompt、上下文或者任务复杂度一变，输出格式就容易失控。

后来我的解决方式是把这套文本协议，改成 LangChain4j 提供的工具调用 API。也就是不再强依赖模型手写 `THOUGHT`、`ACTION`、`OBSERVATION` 这些标签，而是把能力定义成明确的工具，让模型通过标准的 tool call 来表达它要执行什么动作。

调整后的思路是：

- 需要编辑文档、搜索内容、回复用户时，就调用对应工具。
- 如果需要向用户展示中间思考或阶段性结论，就通过 `sendToUser` 这类工具来完成，而不是要求模型硬输出 `THOUGHT` 文本协议。
- 只要模型还没有调用 `terminate` 方法，我就认为当前任务还没有结束，系统继续进入下一轮对话和工具执行。

这样改完之后，整个 Agent 循环稳定了很多，主要收益有三个：

- 工具调用格式由框架约束，解析和执行都更可靠。
- 单轮里“该做什么”由 tool call 明确表达，不容易再出现多个不规范 `ACTION` 混在一起的问题。
- 结束条件也更清晰，是否完成不再依赖模型输出某句特定的话，而是看它有没有调用 `terminate`。

如果在面试里需要一句话总结，我会说：

> 这个问题本质上是“Prompt 约束”不如“结构化工具调用”稳定，所以后来我把 ReAct 的文本协议实现，升级成了基于 LangChain4j tool calling 的执行循环。

## 2. 为什么后来不用大模型直接输出 THOUGHT/ACTION/OBSERVATION，而是改成工具调用？

因为我后来发现，直接让模型输出这套文本协议，本质上还是在“用 Prompt 约束格式”，稳定性依赖模型当下是否听话；而工具调用是“用结构化协议约束行为”，可靠性更高。

对我这个项目来说，工具调用有几个实际好处：

- 模型要执行什么动作，会以标准化的 tool call 表达，后端不需要再从自然语言里硬解析。
- 参数结构更固定，出错时也更容易定位是 Prompt 问题、模型问题还是工具实现问题。
- Agent 的控制权更多回到程序侧，比如什么时候继续下一轮、什么时候结束，不再完全依赖模型输出某几个关键字。

所以我后来的判断是：对于要落地执行的 Agent，Prompt 负责“引导决策”，Tool Calling 负责“约束执行”，这样系统更稳定。

## 3. 你这个 Agent 执行循环是怎么设计的？

我现在的执行循环可以概括成 5 步：

1. 先从 `ChatMemory` 里取出历史消息，加上系统提示词和用户当前指令。
2. 把当前所有可用工具的规范一起发给模型，让模型决定这一轮是否要调用工具。
3. 如果模型发起了 tool call，后端就执行对应工具，并把工具执行结果重新写回上下文。
4. 只要模型还没有调用 `terminate`，我就认为任务还没结束，继续进入下一轮。
5. 如果模型调用了 `terminate`，就把这次任务标记为完成，并返回最终结果。

这样做的好处是，模型负责“决定下一步”，程序负责“控制流程和执行工具”，职责分离比较清楚。

## 4. 你是怎么判断任务结束的？如果模型一直不结束怎么办？

我现在采用的是显式结束机制，也就是只有当模型调用 `terminate` 工具时，我才认定任务完成。

这样设计比让模型输出“done”“task complete”之类的自然语言更可靠，因为：

- 结束条件是结构化信号，不容易误判。
- 可以和普通回复区分开，避免模型只是口头说完成，但实际上没完成关键动作。

同时我也加了 `maxSteps` 这样的上限控制。如果模型一直没有调用 `terminate`，执行循环会在达到最大步数后退出，避免死循环。这其实也是 Agent 系统里很重要的一点：不能把“是否结束”完全交给模型自由发挥，程序侧一定要有兜底机制。

## 5. 工具调用除了稳定性，还有什么工程价值？

我觉得工具调用最大的工程价值，是把“大模型能力”和“系统能力”解耦了。

以前如果靠自然语言约定，模型既要负责理解任务，又要负责产出程序可解析的动作描述，耦合度很高。用了工具调用以后：

- 模型只需要做决策：下一步该调用哪个工具。
- 后端工具负责真正执行：编辑文档、查找内容、给用户发消息、结束任务。
- 每个工具都可以独立测试、独立扩展，后面新增能力时也比较自然。

从工程角度看，这就让 Agent 更像一个“调度层”，而不是把所有逻辑都塞进 Prompt 里。

## 6. 这个项目里，哪些能力是模型负责的，哪些能力是程序负责的？

这是面试里我会主动强调的一点，因为很多 Agent 项目容易把边界讲不清。

我这里的划分是：

- 模型负责理解用户意图、结合上下文做决策、选择合适的下一步动作。
- 程序负责工具注册、参数执行、任务循环控制、状态记录、WebSocket 推送、终止条件和异常兜底。

也就是说，我不会把“流程控制”也交给模型。模型擅长推理和决策，但像循环控制、超时、最大步数、状态管理这些事情，还是应该由程序掌握。这样系统会更可控，也更符合后端工程的设计习惯。

## 7. 如果面试官继续追问这个项目的风险点，你觉得哪些最值得讲？

我觉得有三个点比较值得讲，而且讲出来不会减分，反而能体现工程判断。

第一个是任务状态和异步执行的一致性问题。异步任务场景下，任务对象什么时候创建、什么时候更新、返回给前端的 taskId 和真正执行中的状态对象是否一致，这些都需要设计清楚，否则前端查状态时会出现错位。

第二个是共享状态问题。这个项目现在的文档对象是内存态对象，如果多个任务同时改同一份文档，就会有并发覆盖风险。真正在生产里，我会考虑版本号、乐观锁或者任务串行化。

第三个是上下文和步骤上限控制。Agent 如果没有很清晰的终止机制和步数限制，很容易成本失控，或者在异常情况下反复循环。

所以我会把这个项目定义成：核心链路已经跑通，但如果往生产级演进，优先要补的是状态一致性、并发控制和安全治理。

## 8. 如果让你继续优化这个项目，你会优先做什么？

我会优先做三件事。

第一，把任务状态管理进一步收敛，确保同步执行、异步执行、前端查询状态、WebSocket 推送这几个链路用的是同一套任务标识和状态源。

第二，把文档和任务从纯内存结构升级到持久化存储，至少要保证服务重启后数据不丢，同时为并发控制打基础。

第三，补自动化测试，尤其是 Agent 循环、工具调用、任务终止和异常路径这些关键流程测试。因为 Agent 类项目最怕“看起来能跑，但边界条件一来就失控”，这部分一定要靠测试兜住。

如果用一句话说，就是我下一步不会先追求更多工具，而是先把可控性、可观测性和一致性做好。

## 9. 如果希望支持 rollback Agent 进行的修改，还需要补哪些设计？

如果只做 MVP 级别的回滚，我不会一开始就设计得很复杂，而是先补四个核心能力：执行前快照、任务变更记录、回滚接口、基础一致性校验。

第一，执行前快照。  
在 Agent 任务开始前，先把文档当时的内容保存一份，作为这次任务的 `beforeContent`。这样即使 Agent 后面连续调用了多次编辑工具，我也不需要逐步反推，只要保留任务开始前的原始快照，就能支持最基本的整单回滚。

第二，任务变更记录。  
我会为每次 Agent 执行增加一条任务变更记录，至少包含：

- `taskId`
- `documentId`
- `beforeContent`
- `afterContent`
- `status`
- `createdAt`
- `rolledBackAt`

这样一来，回滚不是“把文档随便改回去”，而是基于某次明确的 Agent 任务做恢复，后续查问题也有依据。

第三，回滚接口。  
我会增加一个类似“按 taskId 回滚”的接口，比如：

- `POST /api/v1/agent/task/{taskId}/rollback`

调用时系统根据这次任务的 `beforeContent`，把文档恢复到执行前状态。对于 MVP 来说，这样已经能覆盖大部分“这次 Agent 改坏了，我想撤销”的场景。

第四，基础一致性校验。  
这是 MVP 里最容易被忽略，但实际上很重要的一点。回滚前我会先校验当前文档内容是否仍然等于这次任务的 `afterContent`。因为如果 Agent 执行完之后，用户又手动改过文档，那这时候直接回滚，就可能把用户的新修改也覆盖掉。

所以 MVP 里我至少会加一个保护逻辑：

- 如果当前内容仍然等于这次任务的 `afterContent`，允许回滚。
- 如果当前内容已经被后续修改过，就拒绝直接回滚，提示存在内容冲突。

如果面试官问我为什么这样设计，我会回答：

> 因为回滚的本质不是“撤销按钮”，而是“把某次任务带来的变更安全地恢复掉”。MVP 阶段最关键的是先把回滚边界定义清楚，避免把后续正常修改一起回滚掉。

如果再用一句话总结这个设计，我会说：

> MVP 回滚不追求按步骤撤销，而是先基于“任务前快照 + 任务后结果 + 冲突校验”实现按任务回滚，这样实现成本低、边界清晰，也最适合作为第一版能力。

## 10. 如果要把 ReAct 和 Planning 模式结合在一起，你会怎么设计？

如果是我来设计，我会采用一个比较清晰的组合方式：先由 Planning Agent 负责“拆任务、定步骤”，再由 ReAct Agent 负责“逐步执行计划”。

这样做的核心原因是，这两种模式擅长的事情不一样：

- Planning 更适合处理复杂任务拆解，先把目标变成有顺序的子任务。
- ReAct 更适合处理执行阶段的不确定性，比如当前应该调哪个工具、工具执行完之后下一步怎么走。

所以我不会让两个模式混在同一层里竞争控制权，而是把它们设计成上下游关系。

### 我会怎么拆职责

第一层是 Planning 层。  
当用户发来一个复杂指令时，先让 Planning Agent 生成一个结构化执行计划，比如：

- 第一步：分析当前文档结构
- 第二步：补充缺失内容
- 第三步：统一格式
- 第四步：生成总结并结束

这里 Planning Agent 的输出，不是直接改文档，而是产出一个 plan，比如 `List<PlanStep>`。

第二层是 ReAct 执行层。  
拿到这个 plan 之后，系统按步骤把子任务依次交给 ReAct Agent。每次只执行当前这一个子任务，ReAct Agent 仍然按照原来的循环方式工作：读取上下文、决定是否调用工具、执行工具、观察结果、直到当前子任务完成。

这样设计之后，Planning 负责“做什么顺序更合理”，ReAct 负责“这一步具体怎么做”。

### 一个比较完整的执行流程

我会把整体流程设计成下面这样：

1. 用户提交复杂编辑任务。
2. 系统先判断这是复杂任务，进入 Planning 阶段。
3. Planning Agent 生成结构化计划。
4. 系统把计划保存到任务状态里，并把当前计划步骤推送给前端。
5. 调度器按顺序取出每个 `PlanStep`，逐个交给 ReAct Agent 执行。
6. 每个 `PlanStep` 执行完成后，更新步骤状态。
7. 如果全部计划步骤完成，再由系统统一结束整单任务。

这样面试里我一般会强调一句：

> Planning 决定“路线图”，ReAct 处理“落地执行中的动态决策”，两者不是替代关系，而是分层协作关系。

### 这样做的好处

我觉得至少有四个明显收益。

第一，复杂任务的可解释性更强。  
如果只靠纯 ReAct，模型可能会直接进入试错式执行，外部看不清它为什么这么做；加上 Planning 之后，前端和用户能先看到整体计划，过程更透明。

第二，执行过程更稳定。  
复杂任务先拆小，每次只让 ReAct 处理一个明确子目标，模型更不容易在单轮里跑偏。

第三，更容易做状态管理和失败恢复。  
因为系统里已经有明确的 `PlanStep`，即使某一步失败，也能知道失败点在哪里，后面也更容易做重试或者局部回滚。

第四，更利于扩展。  
后面如果要加入审批、人工确认、步骤重试、步骤回滚，这些能力都可以挂在 `PlanStep` 这一层，而不是全塞进一个大循环里。

### 如果落到工程设计上，还要补哪些对象

如果我真的往这个方向实现，我至少会增加下面几类对象：

- `Plan`：表示一次任务生成的整体计划
- `PlanStep`：表示计划中的单个步骤，包含步骤内容、顺序、状态
- `PlanStatus`：比如 `PENDING`、`RUNNING`、`COMPLETED`、`FAILED`
- `ExecutionOrchestrator`：负责先跑 Planning，再调度 ReAct 按步骤执行

也就是说，原来的 `AgentState` 主要记录执行状态；结合 Planning 之后，还需要有一层“计划状态”。

### 如果面试官问，为什么不直接让 Planning 自己也去调工具？

我的回答会是：

> 可以，但第一版我不会这样做。因为 Planning 的职责应该尽量保持纯规划，而不是又做规划又做执行。否则很容易重新变成一个边界不清的超级 Agent，最后既不好调试，也不好维护。

所以我的设计原则是：

> Planning 负责把复杂问题拆小，ReAct 负责把每一步做实，调度器负责串起来。

如果再用一句话总结这个方案，我会说：

> 我会把 ReAct 和 Planning 设计成串联关系，而不是并列关系。先让 Planning 产出结构化计划，再让 ReAct 按计划逐步执行，这样复杂任务的稳定性、可解释性和可控性都会更好。

## 11. 如果要把这个项目扩展成多 Agent 系统，你会怎么设计？

如果要把这个项目扩展成多 Agent，我会准备两套讲法。

第一套是我更推荐的方案，也就是 `Supervisor 调度型`。  
第二套是 `协作分工型`，适合在第一套基础上继续扩展。

### 方案一：Supervisor 调度型

这是我更推荐的主方案。核心思路是：系统里有一个 Supervisor Agent，专门负责理解任务、拆分任务、选择合适的子 Agent，并决定执行顺序；其他 Agent 则专注于各自能力边界内的工作。

比如可以设计成下面几类 Agent：

- `PlannerAgent`：负责拆任务和生成执行计划
- `EditorAgent`：负责具体文档编辑
- `RetrieverAgent`：负责检索外部知识或上下文
- `ReviewerAgent`：负责检查结果质量、发现遗漏或风险

这套方案里，Supervisor 不直接做所有事情，而是像一个调度中心：

1. 接收用户请求
2. 判断任务类型和复杂度
3. 选择要调用哪些 Agent
4. 决定这些 Agent 的执行顺序
5. 汇总各 Agent 输出，形成最终结果

如果面试里要一句话概括，我会说：

> Supervisor Agent 负责“统筹”，子 Agent 负责“专精”，这样系统结构更清晰，也更容易扩展。

### 为什么我更推荐 Supervisor 模式

因为它最符合工程上的可控性要求。

第一，职责清楚。  
每个 Agent 都有明确边界，不会演化成一个什么都想做的超级 Agent。

第二，扩展方便。  
后面如果要加新的能力，比如 `ComplianceAgent`、`SummarizerAgent`，只要接入调度体系就行，不需要重写整个主流程。

第三，更容易做监控和治理。  
因为每个任务是被 Supervisor 分派出去的，所以系统更容易记录“谁在什么时机调用了哪个 Agent，产出了什么结果”。

第四，更适合生产化。  
像超时控制、失败重试、熔断、人工兜底这些能力，都更容易挂在 Supervisor 这一层统一治理。

### 方案二：协作分工型

另一种可以讲的方案，是让多个 Agent 以协作方式共同完成任务，而不是由一个中心化 Supervisor 明确调度所有事情。

比如可以设计成：

- `PlannerAgent` 先生成计划
- `EditorAgent` 根据计划编辑文档
- `ReviewerAgent` 对编辑结果做审查
- 如果 Reviewer 发现问题，再把意见返回给 Editor 继续修改

这更像一个“多角色协作流程”，每个 Agent 都像团队里的一个角色，彼此之间会传递中间结果。

这种设计的优点是：

- 更贴近真实团队协作，面试里讲起来比较形象
- 对复杂任务的质量控制会更强，比如天然适合加入审查环节
- 很适合做“生成 - 审核 - 修正”这种闭环

但它的缺点也比较明显：

- Agent 之间的交互链路更长，状态管理更复杂
- 如果没有一个统一协调机制，容易出现互相拉扯、重复工作或者循环讨论
- 调试难度和成本都比单 Supervisor 模式更高

所以我一般会把它定位成第二阶段演进方案，而不是第一版就直接上。

### 如果真要落地，我会增加哪些核心设计

如果从当前这个项目往多 Agent 方向演进，我至少会新增下面几类核心对象：

- `AgentRole`：定义每个 Agent 的角色和职责
- `AgentTask`：表示分派给某个子 Agent 的任务
- `AgentResult`：表示某个 Agent 的输出结果
- `Orchestrator`：负责任务编排、状态推进和结果汇总
- `RoutingStrategy`：决定什么任务该交给哪个 Agent

也就是说，单 Agent 项目里只需要维护一个执行循环；多 Agent 项目里，重点会变成“任务如何拆分、路由、编排、汇总”。

### 我会怎么选第一版落地方案

如果是第一版落地，我一定先做 `Supervisor 调度型`，原因很简单：

- 架构边界更清楚
- 更方便控制复杂度
- 更容易逐步增加新 Agent
- 也更适合和现有单 Agent 架构衔接

我会先把现有的 ReAct Agent、Planning Agent 都保留，然后在外面增加一个 Orchestrator 或 Supervisor 层。这样不是推倒重来，而是在现有能力之上加一层调度。

### 如果面试官追问，多 Agent 最大的挑战是什么

我会回答：最大的挑战不是“多写几个 Agent”，而是“多 Agent 之间怎么协同而不失控”。

具体来说主要有四点：

- 任务边界怎么定义，避免多个 Agent 职责重叠
- 状态怎么管理，避免链路一长就丢上下文
- 调度怎么做，避免出现循环调用或无效协作
- 成本怎么控制，避免 Agent 一多 token 和延迟一起膨胀

所以多 Agent 的核心难点其实不在模型本身，而在编排层设计。

如果最后用一句话总结，我会说：

> 第一版我会采用 Supervisor 调度型多 Agent 架构，把单 Agent 能力沉淀成可复用的专职 Agent；协作分工型适合作为第二阶段演进，用来提升复杂任务的质量和闭环能力。

## 12. 多 Agent 的优势在哪里？

我觉得多 Agent 的核心优势，不是“Agent 越多越高级”，而是当任务复杂度上来之后，系统可以通过分工协作，把一个大而不确定的问题拆成多个边界更清晰的小问题。

如果面试里要先给一句总结，我会说：

> 多 Agent 的价值，本质上在于“分工、协作、解耦和可控性”，它更适合处理复杂任务，而不是简单任务。

### 第一，职责拆分更清楚

单 Agent 模式下，一个 Agent 往往既要理解需求、又要拆任务、又要调用工具、还要检查结果，职责很容易越堆越多。  
多 Agent 模式下，可以把这些能力拆开：

- 一个 Agent 负责规划
- 一个 Agent 负责执行
- 一个 Agent 负责检索
- 一个 Agent 负责审核

这样每个 Agent 的 Prompt、工具集和职责边界都更清晰，整体系统也更容易维护。

### 第二，复杂任务的稳定性更高

复杂任务最怕的是让一个 Agent 一口气做完所有事，因为上下文太长、决策太多，模型很容易跑偏。  
多 Agent 的好处是可以让不同 Agent 只关注自己那一段问题，这样每一跳的任务更明确，错误范围也更小。

比如：

- Planner 只负责把任务拆清楚
- Editor 只负责根据子任务改文档
- Reviewer 只负责检查结果质量

这比让一个 Agent 同时承担所有角色更稳定。

### 第三，结果质量更容易做闭环

单 Agent 做完之后，往往只能相信它“自己觉得自己完成了”。  
多 Agent 体系里，可以天然加入复核机制，比如生成之后再审核、审核后再修正，这样质量闭环会更自然。

这在面试里是很值得讲的一点，因为它说明多 Agent 不是为了炫技，而是为了让系统具备“互相制衡”和“二次检查”的能力。

### 第四，更容易扩展新能力

如果后续要增加新能力，单 Agent 往往意味着继续改一个越来越重的 Prompt 和越来越复杂的工具集合。  
多 Agent 模式下，可以直接新增一个专职 Agent，然后接入编排层。

比如未来如果要加：

- `ComplianceAgent` 做合规检查
- `SummaryAgent` 做结果总结
- `TranslatorAgent` 做多语言处理

都不一定要改原有 Agent 的核心逻辑，只要接到调度链路里就可以。

### 第五，更利于治理和观测

从工程角度看，多 Agent 还有一个很大的优势，就是更容易做治理。

因为任务被拆成多个节点之后，系统能更清楚地看到：

- 哪个 Agent 在执行
- 每一步花了多少时间
- 哪一步失败了
- 哪个 Agent 消耗了更多 token

这对于后续做监控、优化成本、分析瓶颈都很有帮助。  
而单 Agent 很容易变成一个黑盒，最后只知道“它结果不太对”，但不知道是哪个阶段出了问题。

### 但我也会主动补一句 trade-off

我一般不会把多 Agent 说成银弹。我会主动补一句：

> 多 Agent 的前提是任务确实足够复杂，值得为分工付出额外编排成本。对于简单任务，单 Agent 往往更直接、更便宜。

这样回答会显得更成熟，因为你不是机械地说“多 Agent 一定更好”，而是知道它的适用边界。

如果最后再用一句话总结，我会说：

> 多 Agent 最大的优势，是把复杂任务拆成多个可控环节，让系统在稳定性、质量闭环、扩展性和治理能力上都更强；但前提是任务复杂度足够高，值得引入这层编排成本。
